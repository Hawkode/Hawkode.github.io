Guide

Patch Management Workflow

This guide explains a practical patch management workflow for a lab or small IT environment. It covers checking patch status, identifying missing updates, prioritising risk, planning the change, applying patches, validating results, and recording evidence.

Patch management is not only about installing updates. A good process includes asset tracking, testing, rollback planning, validation, and clear documentation.

Lab Setup

Example environment

This example uses a small lab with one Windows machine, one Ubuntu server, and a simple patch tracker.

  • Windows target: Windows 10 or Windows 11 test machine
  • Linux target: Ubuntu Server test machine
  • Patch sources: Windows Update and Ubuntu package repositories
  • Validation tools: PowerShell, terminal commands, and vulnerability scan results
  • Tracking: spreadsheet, markdown notes, or ticket-style patch tracker

Only test patching on systems you own or have permission to manage.

Step 1

Create an asset inventory

Start by listing the systems in scope. Patch decisions are easier when each device has clear details, ownership, exposure, and priority.

Example Windows asset record:

Asset ID: LAB-001
Hostname: LAB-WIN11
IP Address: 192.168.56.20
Operating System: Windows 11
Owner: Lab
Exposure: Internal lab only
Criticality: Medium
Patch Group: Workstations

Example Linux asset record:

Asset ID: LAB-002
Hostname: LAB-UBUNTU01
IP Address: 192.168.56.30
Operating System: Ubuntu Server
Owner: Lab
Exposure: Internal lab only
Criticality: Medium
Patch Group: Linux servers

Step 2

Check the current patch status

Check the current update state before making changes. This gives you a baseline and helps prove what changed after patching.

Windows PowerShell checks:

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Ubuntu checks:

lsb_release -a

uname -r

apt list --upgradable

Save the output before patching. This can be used as evidence later.

Note: Get-HotFix is useful, but it may not show every Windows update type. Windows Update history, endpoint management tools, or vulnerability scanner results may also be needed.

Step 3

Identify missing updates

Missing updates can be found through operating system tools, package managers, vulnerability scanners, vendor advisories, and patch management platforms.

Ubuntu update check:

sudo apt update

apt list --upgradable

Example missing update record:

Asset: LAB-UBUNTU01
Package: openssl
Current Version: Older installed version
Available Update: Newer security update available
Source: apt list --upgradable
Risk: Security update pending
Status: Open

For security updates, record the affected software, version, CVE if available, severity, affected asset, and source of the finding.

Step 4

Prioritise patches by risk

Not every update has the same priority. Rank patches based on severity, exploit activity, asset importance, exposure, and disruption risk.

  • Urgent: exploited vulnerability on an internet-facing system
  • High: security update for an important internal server
  • Medium: standard monthly security update for a workstation
  • Low: non-security update with low impact

Example priority decision:

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Exposure: Internal lab only
Asset Criticality: Medium
Severity: High
Disruption Risk: Low
Priority: High
Action: Patch during next lab maintenance window

Step 5

Plan the change

Patch deployment should be planned, especially on important systems. A change plan helps reduce downtime and gives a clear recovery path.

Example change plan:

Change ID: CHG-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Reason: OpenSSL update available
Maintenance Window: Saturday 12:00-14:00
Expected Impact: Reboot may be required
Rollback Plan: Restore VM snapshot if the update causes issues
Validation: Confirm package version and run follow-up checks
Owner: Lab

Before patching, check:

  • The asset is in scope.
  • A backup or snapshot is available.
  • A rollback plan exists.
  • The maintenance window is clear.
  • Reboot impact has been considered.

Step 6

Back up or snapshot the system

Before applying patches, take a snapshot or backup where possible. This is useful in a virtual lab and important in production environments.

Example snapshot record:

Asset: LAB-UBUNTU01
Snapshot Name: pre-patch-openssl
Reason: Rollback point before security update
Status: Completed

In a lab, a VM snapshot is usually enough. In a business environment, follow the approved backup and change process.

Step 7

Apply the patch

Apply the update during the planned maintenance window. Record the commands used, the result, and any errors.

Ubuntu patch commands:

sudo apt update

sudo apt upgrade

Check if a reboot is required:

test -f /var/run/reboot-required && cat /var/run/reboot-required

Windows systems can be patched through Windows Update, Microsoft Intune, WSUS, Microsoft Configuration Manager, or another approved patching tool.

Example deployment record:

Asset: LAB-UBUNTU01
Patch Window: Saturday 12:00-14:00
Command Used: sudo apt update && sudo apt upgrade
Result: Updates installed successfully
Reboot Required: Yes
Reboot Completed: Yes
Issues Found: None
Status: Patched

Step 8

Validate the result

After patching, confirm that the update installed correctly and the system still works. Do not mark the patch as complete until validation is done.

Ubuntu validation examples:

apt list --upgradable

dpkg -l | grep openssl

uname -r

systemctl --failed

Windows validation examples:

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber

If a vulnerability scanner found the original issue, run a follow-up scan and compare the before and after results.

Example validation record:

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Validation Method: Package check and vulnerability rescan
Result: Update installed and finding no longer appears
Evidence: dpkg output and scanner report saved
Status: Validated

Step 9

Handle failed or deferred patches

Some patches fail or need to be delayed. They should still be tracked with a reason, risk owner, temporary control, and review date.

Common patch issues:

  • Update fails to install.
  • System requires a reboot.
  • Service fails after patching.
  • Application compatibility issue appears.
  • Disk space is too low.
  • Network or repository issue blocks the update.

Example deferred patch record:

Patch Item: Application security update
Asset: LAB-SERVER01
Reason Not Applied: Application compatibility issue
Temporary Control: Restricted access to internal subnet
Owner: Lab
Review Date: 2026-07-27
Status: Deferred

Step 10

Update the patch tracker

A patch tracker gives a clear view of open, scheduled, patched, failed, deferred, and validated updates.

Example tracker entry:

ID: PATCH-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Package or KB: OpenSSL update
Priority: High
Owner: Lab
Status: Validated
Validation Evidence: dpkg output and vulnerability rescan
Notes: No issues after reboot

Useful patch status values:

  • Open: patch identified but not planned yet
  • Scheduled: patch planned for a maintenance window
  • Patched: patch installed but not yet validated
  • Validated: patch confirmed and system checked
  • Failed: patch did not install or caused an issue
  • Deferred: patch delayed with a documented reason

Evidence

Recommended evidence to save

  • Asset inventory
  • Pre-patch operating system version
  • List of missing updates
  • Patch priority decision
  • Change plan or patch notes
  • Snapshot or backup record
  • Patch command output or screenshots
  • Reboot confirmation if required
  • Post-patch version check
  • Service health checks
  • Follow-up vulnerability scan result
  • Patch tracker entry

Common Mistakes

Things to avoid

  • Patching without knowing which systems are in scope.
  • Skipping backups or snapshots before risky updates.
  • Installing patches without checking reboot requirements.
  • Marking patches as complete without validation.
  • Ignoring failed patches.
  • Delaying patches without recording the reason.
  • Forgetting to compare before and after scan results.

Summary

Quick workflow

  1. Create an asset inventory.
  2. Check the current patch status.
  3. Identify missing updates.
  4. Prioritise patches by risk.
  5. Plan the change.
  6. Back up or snapshot the system.
  7. Apply the patch.
  8. Validate the result.
  9. Handle failed or deferred patches.
  10. Update the patch tracker.
  11. Save evidence.

Why It Matters

Why patch management matters in IT and security

Patch management reduces risk by fixing known weaknesses before they can be abused. It also supports system stability, change control, vulnerability management, and reliable IT operations.

In real environments, patching is not just pressing update. It involves inventory, testing, risk decisions, rollback planning, validation, and reporting.
