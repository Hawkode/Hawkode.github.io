Patch Management Workflow

This guide shows a practical patch management workflow for a lab or small environment. The goal is to identify missing updates, prioritise patches by risk, plan the change, test the update, deploy it, validate the result, and document the work.

Patch management is closely linked to vulnerability management. Vulnerability scanning helps identify risk. Patch management turns that information into controlled action.

Lab Setup

Example environment

This lab uses a simple test environment with one Windows device, one Linux device, and a basic patch tracker.

  • Windows target: Windows 10 or Windows 11 test machine
  • Linux target: Ubuntu Server test machine
  • Patch sources: Windows Update and Ubuntu package repositories
  • Validation tools: PowerShell, terminal commands, and optional vulnerability scanner
  • Tracking: spreadsheet, markdown file, or ticket-style notes

Only test patching on systems you own or have permission to manage.

Step 1

Create an asset and patch inventory

Start by listing the systems you are responsible for. Patch decisions are easier when each device has clear ownership, operating system details, and importance.

Example asset record:

Asset ID: LAB-001
Hostname: LAB-WIN11
IP Address: 192.168.56.20
Operating System: Windows 11
Owner: Lab
Exposure: Internal lab only
Criticality: Medium
Patch Group: Workstations
Maintenance Window: Saturday 10:00-12:00

Example Linux asset record:

Asset ID: LAB-002
Hostname: LAB-UBUNTU01
IP Address: 192.168.56.30
Operating System: Ubuntu Server
Owner: Lab
Exposure: Internal lab only
Criticality: Medium
Patch Group: Linux servers
Maintenance Window: Saturday 12:00-14:00

Step 2

Check current patch status

Check the current update state before making changes. This gives you a baseline and helps prove what changed after patching.

Windows PowerShell examples:

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber
Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Ubuntu examples:

lsb_release -a
uname -r
apt list --upgradable

Save the output before patching. This becomes part of your evidence.

Step 3

Identify missing updates

Missing updates can come from operating system checks, vulnerability scanners, vendor advisories, or package manager output.

Windows options:

  • Windows Update settings
  • PowerShell update modules in a lab
  • Microsoft Update history
  • Vulnerability scanner results

Ubuntu command:

sudo apt update
apt list --upgradable

Example finding:

Asset: LAB-UBUNTU01
Package: openssl
Current Version: Older installed version
Available Update: Newer security update available
Source: apt list --upgradable
Risk: Security update pending
Status: Open

Step 4

Prioritise patches by risk

Not every patch has the same priority. Rank updates based on exposure, severity, asset importance, and chance of disruption.

Example priority model:

  • Urgent: exploited vulnerability on an internet-facing system
  • High: security patch for a business-critical internal server
  • Medium: normal monthly security update for an internal workstation
  • Low: non-security update with low operational impact

Example decision:

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Exposure: Internal lab only
Asset Criticality: Medium
Severity: High
Disruption Risk: Low
Priority: High
Action: Patch during next lab maintenance window

A security update for a public-facing server should normally take priority over a feature update on an isolated test machine.

Step 5

Plan the change

Patch deployment should be controlled. Even in a lab, write a short change plan. This shows that you understand risk, timing, testing, and rollback.

Example change plan:

Change ID: CHG-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Reason: OpenSSL update available
Planned Date: 2026-06-27
Maintenance Window: 12:00-14:00
Expected Impact: Short service restart may be required
Backout Plan: Restore VM snapshot if update causes issues
Validation: Confirm package version and rerun vulnerability scan
Owner: Lab

Before patching, check:

  • Is the asset in scope?
  • Is there a backup or snapshot?
  • Is there a rollback plan?
  • Is the maintenance window agreed?
  • Could the update restart a service?
  • Could the update require a reboot?

Step 6

Back up or snapshot the system

Before applying patches, take a snapshot or backup where possible. This is especially useful in a virtual lab.

Example snapshot record:

Asset: LAB-UBUNTU01
Snapshot Name: pre-patch-openssl-2026-06-27
Snapshot Time: 11:55
Reason: Rollback point before security updates
Status: Completed

For a real environment, the backup method depends on the system and business process. For a lab, a VM snapshot is usually enough.

Step 7

Test the patch

Where possible, test patches before applying them to important systems. In a lab, this can mean patching one test machine first and checking for issues.

Basic test checks:

  • Does the system boot normally after patching?
  • Can you still log in?
  • Are key services running?
  • Are expected ports still open or closed?
  • Did the patch introduce errors?

Ubuntu service check example:

systemctl --failed

Windows service check example:

Get-Service | Where-Object {$_.Status -eq "Stopped"} | Select-Object Name, DisplayName, Status

Step 8

Deploy the patch

Apply the update during the planned window. Record the commands used, the result, and any errors.

Ubuntu update commands:

sudo apt update
sudo apt upgrade

To apply only security updates, the method depends on how the system is configured. In a lab, a normal package upgrade is acceptable for learning and documentation.

Check if a reboot is required on Ubuntu:

cat /var/run/reboot-required

Windows update options:

  • Use Windows Update in Settings
  • Use managed patching tools in business environments
  • Use PowerShell modules in a lab if configured

Example deployment record:

Asset: LAB-UBUNTU01
Patch Window: 12:00-14:00
Command Used: sudo apt update && sudo apt upgrade
Result: Updates installed successfully
Reboot Required: Yes
Reboot Completed: Yes
Issues Found: None
Status: Patched

Step 9

Validate the update

After patching, confirm that the update was applied and the system still works. Do not mark the patch as complete until validation is done.

Ubuntu validation examples:

apt list --upgradable
dpkg -l | grep openssl
uname -r

Windows validation examples:

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10
Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber

Optional scanner validation:

  • Run a follow-up vulnerability scan.
  • Compare the before and after results.
  • Confirm the original finding is gone or reduced.
  • Save the scan result as evidence.

Example validation record:

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Validation Method: Package check and vulnerability rescan
Result: Update installed and finding no longer appears
Evidence: dpkg output and scanner report saved
Status: Validated
Closed Date: 2026-06-27

Step 10

Handle failed patches

Patches can fail. The important part is recording what happened and taking a controlled next step.

Common patch issues:

  • Update fails to install
  • System requires a reboot
  • Service fails after patching
  • Application compatibility issue
  • Disk space problem
  • Network or repository issue

Example failed patch record:

Patch Item: Security update
Asset: LAB-WIN11
Issue: Update failed during installation
Error: Installation did not complete
Impact: Patch not applied
Action Taken: Rebooted device and retried update
Result: Patch installed successfully on second attempt
Status: Validated

If a patch cannot be applied, document the reason and add a temporary control. 

Patch Item: Application security update
Asset: LAB-SERVER01
Reason Not Applied: Application compatibility issue
Temporary Control: Restricted access to internal subnet
Owner: Lab
Review Date: 2026-07-27
Status: Deferred

Step 11

Update the patch tracker

A patch tracker gives a clear view of what is open, planned, patched, failed, deferred, or validated.

Example tracker entry:

ID: PATCH-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Package or KB: OpenSSL update
Priority: High
Owner: Lab
Planned Date: 2026-06-27
Status: Validated
Validation Evidence: dpkg output and vulnerability rescan
Notes: No issues after reboot

Useful patch status values:

  • Open: Patch identified but not planned yet.
  • Scheduled: Patch planned for a maintenance window.
  • In Progress: Patch deployment has started.
  • Patched: Patch installed but not yet validated.
  • Validated: Patch confirmed and system checked.
  • Failed: Patch did not install or caused an issue.
  • Deferred: Patch delayed with a documented reason.

Step 12

Write a patch report

A good patch report should explain what was updated, what failed, what still needs work, and what risk remains.

Example patch report:

Patch Report: June Lab Patch Cycle
Date: 2026-06-27
Scope: LAB-WIN11, LAB-UBUNTU01
Patch Window: 10:00-14:00

Summary:
Total Assets Reviewed: 2
Assets Patched: 2
Patches Applied: 8
Failed Patches: 0
Deferred Patches: 0
Reboots Required: 2

Completed:
- LAB-WIN11 Windows updates applied and validated
- LAB-UBUNTU01 package updates applied and validated

Validation:
- Patch status checked after reboot
- Key services checked
- Follow-up vulnerability scan completed

Issues:
- No issues found

Next Actions:
- Continue monthly patch cycle
- Review deferred patches if any appear
- Keep asset inventory updated

Evidence

Recommended evidence to save

  • Asset inventory
  • Pre-patch operating system version
  • List of missing updates
  • Patch plan or change record
  • Snapshot or backup record
  • Patch command output or screenshots
  • Reboot confirmation if required
  • Post-patch version check
  • Service health checks
  • Follow-up vulnerability scan result
  • Patch tracker entry
  • Final patch report

Common Mistakes

Things to avoid

  • Patching without knowing which systems are in scope.
  • Skipping backups or snapshots before risky updates.
  • Installing patches without checking for reboot requirements.
  • Marking patches as complete without validation.
  • Ignoring failed patches.
  • Delaying patches without recording the reason.
  • Relying only on screenshots and not keeping structured notes.
  • Forgetting to compare before and after scan results.

Summary

Quick workflow

  1. Create an asset and patch inventory.
  2. Check the current patch status.
  3. Identify missing updates.
  4. Prioritise patches by risk.
  5. Plan the change.
  6. Back up or snapshot the system.
  7. Test the patch where possible.
  8. Deploy the patch.
  9. Validate the update.
  10. Handle failed or deferred patches.
  11. Update the patch tracker.
  12. Write a patch report.

Why It Matters

Why patch management matters in IT and security

Patch management reduces risk by fixing known weaknesses before they can be abused. It also supports system stability, compliance, and good operational practice.

In real environments, patching is not just pressing update. It involves inventory, testing, risk decisions, change control, rollback planning, validation, and reporting.
