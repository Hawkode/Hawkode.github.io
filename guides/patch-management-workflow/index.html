Guide

Patch Management Workflow

This guide shows a practical patch management workflow for a small lab environment. The aim is to check patch status, identify missing updates, prioritise risk, apply patches, validate the result, and save clear evidence.

In a real IT environment, patching is not just pressing update. It includes asset tracking, risk decisions, testing, rollback planning, validation, and documentation.

Lab goal: complete a full patch cycle from discovery to validation.

Output: asset record, missing update list, patch plan, validation result, and evidence notes.

Lab Setup

Example environment

This lab uses one Windows endpoint and one Ubuntu server. The same process can be scaled to more systems.

Asset 1 

Asset ID: LAB-001
Hostname: LAB-WIN11
IP Address: 192.168.56.20
Operating System: Windows 11
Role: Test workstation
Exposure: Internal lab only
Criticality: Medium

Asset 2 

Asset ID: LAB-002
Hostname: LAB-UBUNTU01
IP Address: 192.168.56.30
Operating System: Ubuntu Server
Role: Test server
Exposure: Internal lab only
Criticality: Medium

Step 1

Check the current patch status

Start by checking the current system version and installed updates. This gives you a baseline before making changes.

Windows checks 

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Ubuntu checks 

lsb_release -a

uname -r

apt list --upgradable

Evidence to save: OS version, kernel/build number, installed update list, and available update list.

Step 2

Identify missing updates

Missing updates can come from Windows Update, Linux package managers, vulnerability scanners, vendor advisories, or endpoint management tools.

For Ubuntu, refresh the package list and check what can be upgraded. 

sudo apt update

apt list --upgradable

Example finding:

Finding ID: PATCH-001
Asset: LAB-UBUNTU01
Package: openssl
Finding Source: apt list --upgradable
Issue: Security update available
Status: Open

For Windows, compare Windows Update, installed hotfixes, and any scanner findings.

Note: Get-HotFix is useful, but it may not show every Windows update type. Use Windows Update history or management tool reports when needed.

Step 3

Prioritise the patch

Do not treat every patch the same. Priority should be based on risk.

Urgent — exploited vulnerability on an internet-facing system.

High — security update for an important internal server.

Medium — standard monthly security update for a workstation.

Low — non-security update with low impact.

Example priority decision:

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Exposure: Internal lab only
Asset Criticality: Medium
Severity: High
Disruption Risk: Low
Priority: High
Action: Patch during next maintenance window

Step 4

Plan the change

Before applying the patch, write a short change plan. This shows what will change, when it will happen, and how you will recover if something breaks. 

Change ID: CHG-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Reason: OpenSSL update available
Maintenance Window: Saturday 12:00-14:00
Expected Impact: Reboot may be required
Rollback Plan: Restore VM snapshot
Validation Method: Confirm package version and check services
Owner: Lab

Change checks:

  • Asset is in scope.
  • Maintenance window is clear.
  • Snapshot or backup is available.
  • Rollback method is known.
  • Validation steps are ready.

Step 5

Take a snapshot or backup

Before patching, create a rollback point. In a virtual lab, this is normally a VM snapshot. 

Asset: LAB-UBUNTU01
Snapshot Name: pre-patch-openssl
Reason: Rollback point before security update
Status: Completed

In a business environment, follow the approved backup and change process.

Step 6

Apply the patch

Apply the patch during the planned maintenance window and record what happened.

Ubuntu patch commands 

sudo apt update

sudo apt upgrade

Check if a reboot is required: 

test -f /var/run/reboot-required && cat /var/run/reboot-required

Windows patching options

  • Windows Update
  • Microsoft Intune
  • WSUS
  • Microsoft Configuration Manager
  • Approved business patching tool

Example deployment record:

Asset: LAB-UBUNTU01
Patch Window: Saturday 12:00-14:00
Command Used: sudo apt update && sudo apt upgrade
Result: Updates installed successfully
Reboot Required: Yes
Reboot Completed: Yes
Issues Found: None
Status: Patched

Step 7

Validate the result

Validation confirms the update installed correctly and the system still works. Do not close the patch item until this is done.

Ubuntu validation 

apt list --upgradable

dpkg -l | grep openssl

uname -r

systemctl --failed

Windows validation 

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber

Example validation record:

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Validation Method: Package check and service check
Result: Update installed and no failed services found
Evidence: dpkg output and systemctl output saved
Status: Validated

Validated means: the patch is installed, the system is stable, key services work, and evidence has been saved.

Step 8

Handle failed or delayed patches

Failed and delayed patches should stay visible until they are fixed, accepted as risk, or replaced by another control.

Common issues:

  • Patch fails to install.
  • System needs a reboot.
  • Service fails after patching.
  • Application compatibility issue appears.
  • Disk space is too low.
  • Repository or network issue blocks the update.

Example deferred patch record:

Patch Item: Application security update
Asset: LAB-SERVER01
Reason Not Applied: Compatibility issue
Temporary Control: Restricted access to internal subnet
Owner: Lab
Review Date: 2026-07-27
Status: Deferred

Step 9

Update the patch tracker

A patch tracker gives a clear view of what is open, scheduled, patched, validated, failed, or deferred. 

ID: PATCH-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Package or KB: OpenSSL update
Priority: High
Owner: Lab
Status: Validated
Evidence: dpkg output and service check
Notes: No issues after reboot

Useful status values:

  • Open: patch identified but not planned yet.
  • Scheduled: patch planned for a maintenance window.
  • Patched: patch installed but not validated yet.
  • Validated: patch confirmed and system checked.
  • Failed: patch did not install or caused an issue.
  • Deferred: patch delayed with a recorded reason.

Evidence

Evidence to save

Save enough evidence to prove what was checked, what was patched, and what was validated.

  • Asset inventory entry
  • Pre-patch OS version
  • Pre-patch update list
  • Missing update finding
  • Patch priority decision
  • Change plan or patch notes
  • Snapshot or backup record
  • Patch command output
  • Reboot confirmation if required
  • Post-patch version check
  • Service health check
  • Follow-up vulnerability scan result if used
  • Patch tracker entry

Common Mistakes

Things to avoid

  • Patching without knowing which assets are in scope.
  • Skipping snapshots before risky updates.
  • Ignoring reboot requirements.
  • Marking patches complete without validation.
  • Not recording failed patches.
  • Delaying patches without a reason or review date.
  • Saving screenshots but no structured notes.
  • Not comparing before and after results.

Summary

Quick workflow

  1. List the assets in scope.
  2. Check the current patch status.
  3. Identify missing updates.
  4. Prioritise patches by risk.
  5. Plan the change.
  6. Take a snapshot or backup.
  7. Apply the patch.
  8. Validate the result.
  9. Record failed or delayed patches.
  10. Update the patch tracker.
  11. Save evidence.

Why It Matters

Why patch management matters

Patch management reduces the risk of known vulnerabilities being abused. It also supports system stability, change control, vulnerability management, and reliable IT operations.

This lab shows the full patch cycle: finding missing updates, planning the change, applying patches, validating the result, and documenting the evidence.
