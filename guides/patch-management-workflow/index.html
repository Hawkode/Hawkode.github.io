Guide

Patch Management Workflow

A practical workflow for checking missing updates, planning patch changes, applying fixes, validating systems, and documenting the result.

This guide follows a small lab setup with one Windows device, one Linux device, and a simple patch tracker.

Lab Setup

Example environment

Use a small lab so the patching process is safe and easy to document.

  • Windows target: Windows 10 or Windows 11 test machine
  • Linux target: Ubuntu Server test machine
  • Patch sources: Windows Update and Ubuntu repositories
  • Validation: PowerShell, terminal checks, and optional vulnerability scan
  • Tracking: spreadsheet, markdown notes, or a ticket-style tracker

Only patch systems you own or have permission to manage.

Step 1

Create an asset inventory

Start by recording the systems in scope. This makes it easier to plan, prioritise, and validate patching work. 

Asset ID: LAB-001
```

Hostname: LAB-WIN11
IP Address: 192.168.56.20
Operating System: Windows 11
Owner: Lab
Exposure: Internal lab only
Criticality: Medium
``` 
Asset ID: LAB-002
```

Hostname: LAB-UBUNTU01
IP Address: 192.168.56.30
Operating System: Ubuntu Server
Owner: Lab
Exposure: Internal lab only
Criticality: Medium
```

Step 2

Check current patch status

Check the current update state before making changes. This gives you a baseline for before and after comparison.

Windows examples:

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber
Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Ubuntu examples:

lsb_release -a
```

uname -r
apt list --upgradable
```

Step 3

Identify missing updates

Missing updates can come from Windows Update, package manager output, vulnerability scanner results, or vendor advisories.

Ubuntu check:

sudo apt update
```

apt list --upgradable
```

Example finding:

Asset: LAB-UBUNTU01
```

Package: openssl
Source: apt list --upgradable
Risk: Security update pending
Status: Open
```

Step 4

Prioritise patches

Not every patch has the same priority. Rank patches based on severity, exposure, asset importance, and disruption risk.

  • Urgent: exploited vulnerability on an internet-facing system
  • High: security patch for an important internal server
  • Medium: normal monthly security update
  • Low: non-security update with low impact
Patch Item: OpenSSL security update
```

Asset: LAB-UBUNTU01
Exposure: Internal lab only
Severity: High
Disruption Risk: Low
Priority: High
Action: Patch during the next maintenance window
```

Step 5

Plan the change

Before patching, write a short change plan. This shows the reason, timing, risk, rollback option, and validation method. 

Change ID: CHG-001
```

Asset: LAB-UBUNTU01
Patch Type: Security update
Reason: OpenSSL update available
Maintenance Window: Saturday 12:00-14:00
Expected Impact: Possible service restart or reboot
Rollback: Restore VM snapshot if needed
Validation: Confirm package version and run checks
```

Before patching, check:

  • Is the asset in scope?
  • Is there a backup or snapshot?
  • Is there a rollback plan?
  • Could the patch require a reboot?
  • Could the patch affect a service?

Step 6

Back up or snapshot

Take a backup or VM snapshot before applying patches, especially when testing updates that could affect system stability. 

Asset: LAB-UBUNTU01
```

Snapshot Name: pre-patch-2026-06-27
Reason: Rollback point before updates
Status: Completed
```

Step 7

Deploy the patch

Apply the update during the planned window and record what was done.

Ubuntu update:

sudo apt update
```

sudo apt upgrade
```

Check if Ubuntu needs a reboot:

cat /var/run/reboot-required

Windows options:

  • Use Windows Update in Settings.
  • Use managed patching tools in a business environment.
  • Use PowerShell modules in a lab if configured.
Asset: LAB-UBUNTU01
```

Command Used: sudo apt update && sudo apt upgrade
Result: Updates installed successfully
Reboot Required: Yes
Status: Patched
```

Step 8

Validate the update

After patching, confirm the update was applied and the system still works. Do not mark a patch as complete without validation.

Ubuntu validation:

apt list --upgradable
```

dpkg -l | grep openssl
systemctl --failed
```

Windows validation:

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Optional validation:

  • Run a follow-up vulnerability scan.
  • Compare before and after results.
  • Confirm the original finding is gone or reduced.
  • Save the validation output as evidence.

Step 9

Handle failed or deferred patches

If a patch fails or must be delayed, document the issue, the risk, the temporary control, and the next review date. 

Patch Item: Application security update
```

Asset: LAB-SERVER01
Reason Not Applied: Application compatibility issue
Temporary Control: Restricted access to internal subnet
Owner: Lab
Review Date: 2026-07-27
Status: Deferred
```

Step 10

Update the patch tracker

A patch tracker gives a clear record of what is open, planned, patched, failed, deferred, or validated. 

ID: PATCH-001
```

Asset: LAB-UBUNTU01
Patch Type: Security update
Priority: High
Status: Validated
Validation Evidence: Package check and scan result
Notes: No issues after reboot
```

Useful status values:

  • Open: Patch identified but not planned yet.
  • Scheduled: Patch planned for a maintenance window.
  • Patched: Patch installed but not yet validated.
  • Validated: Patch confirmed and system checked.
  • Failed: Patch did not install or caused an issue.
  • Deferred: Patch delayed with a documented reason.

Step 11

Write a patch report

A patch report should explain what was patched, what failed, what was deferred, and what still needs attention. 

Patch Report: Lab Patch Cycle
```

Scope: LAB-WIN11, LAB-UBUNTU01

Summary:
Assets Reviewed: 2
Assets Patched: 2
Failed Patches: 0
Deferred Patches: 0
Reboots Required: 2

Validation:

* Patch status checked after reboot
* Key services checked
* Follow-up scan completed

Next Actions:

* Continue monthly patch cycle
* Review deferred patches
* Keep asset inventory updated

Evidence

Recommended evidence to save

```
  • Asset inventory
  • Pre-patch version checks
  • Missing updates list
  • Change plan
  • Snapshot or backup record
  • Patch command output or screenshots
  • Post-patch validation output
  • Follow-up scan result
  • Patch tracker entry
  • Final patch report
```

Common Mistakes

Things to avoid

```
  • Patching without knowing which systems are in scope.
  • Skipping backups or snapshots before risky updates.
  • Forgetting to check reboot requirements.
  • Marking patches as complete without validation.
  • Ignoring failed patches.
  • Delaying patches without recording the reason.
  • Forgetting to compare before and after results.
```

Summary

Quick workflow

```
  1. Create an asset inventory.
  2. Check current patch status.
  3. Identify missing updates.
  4. Prioritise patches by risk.
  5. Plan the change.
  6. Back up or snapshot the system.
  7. Deploy the patch.
  8. Validate the update.
  9. Handle failed or deferred patches.
  10. Update the patch tracker.
  11. Write a patch report.
```

Why It Matters

Why patch management matters

```

Patch management reduces risk by fixing known weaknesses before they can be abused. It also supports system stability, security operations, and good IT practice.

In real environments, patching is not just pressing update. It involves inventory, testing, risk decisions, rollback planning, validation, and reporting.
