Security Guide

Patch Management Workflow

This guide shows a practical patch management workflow for a small lab environment. The goal is to check patch status, identify missing updates, prioritise risk, apply patches, validate the result, and save evidence.

Patch management is not only about installing updates. A proper workflow includes asset tracking, risk decisions, rollback planning, testing, validation, and documentation.

Lab Goal

Complete a full patch cycle from update discovery to validation.

Final Output

Asset notes, missing update records, patch plan, validation output, and evidence.

Step 1

Check Current Patch Status

Start by checking the current system version, installed updates, and available updates. This gives you a baseline before making changes.

Windows Checks

PowerShell · Build Number · Hotfixes

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Note: Get-HotFix is useful, but it may not show every Windows update type. Windows Update history or endpoint management reports may also be needed.

Ubuntu Checks

Terminal · Kernel · Upgradable Packages

lsb_release -a

uname -r

apt list --upgradable

Save this output before patching. It helps prove what changed after the update.

Step 2

Identify Missing Updates

Missing updates can be found using Windows Update, Linux package managers, vendor advisories, vulnerability scanners, or patch management tools.

Ubuntu Update Check

APT · Package Updates · Security Fixes

sudo apt update

apt list --upgradable

Example Finding

Finding Record · Patch Tracker

Finding ID: PATCH-001
Asset: LAB-UBUNTU01
Package: openssl
Finding Source: apt list --upgradable
Issue: Security update available
Risk: Security update pending
Status: Open

For security updates, record the affected software, version, CVE if available, severity, affected asset, and source of the finding.

Step 3

Prioritise Patches By Risk

Not every patch has the same priority. Rank patches based on severity, exploit activity, asset importance, exposure, and disruption risk.

Urgent

Exploited vulnerability on an internet-facing system.

High

Security update for an important internal server.

Medium

Standard monthly security update for a workstation.

Low

Non-security update with low operational impact.

Example Priority Decision

Risk Based Patching

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Exposure: Internal lab only
Asset Criticality: Medium
Severity: High
Disruption Risk: Low
Priority: High
Action: Patch during next maintenance window

Step 4

Plan The Change

Before applying the patch, write a short change plan. This shows what will change, when it will happen, what the impact could be, and how rollback would work.

Example Change Plan

Change Planning · Rollback · Validation

Change ID: CHG-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Reason: OpenSSL update available
Maintenance Window: Saturday 12:00-14:00
Expected Impact: Reboot may be required
Rollback Plan: Restore VM snapshot
Validation Method: Confirm package version and check services
Owner: Lab

Before Patching

  • Confirm the asset is in scope.
  • Check the maintenance window.
  • Confirm rollback method.
  • Prepare validation checks.

Possible Impact

  • Service restart
  • System reboot
  • Temporary downtime
  • Application compatibility issue

Step 5

Take A Snapshot Or Backup

Before patching, create a rollback point. In a virtual lab, this is normally a VM snapshot.

Example Snapshot Record

Rollback Point · VM Snapshot

Asset: LAB-UBUNTU01
Snapshot Name: pre-patch-openssl
Reason: Rollback point before security update
Status: Completed

In a business environment, follow the approved backup and change process.

Step 6

Apply The Patch

Apply the patch during the planned maintenance window and record what happened.

Ubuntu Patch Commands

APT · Update · Upgrade

sudo apt update

sudo apt upgrade

Check if a reboot is required: 

test -f /var/run/reboot-required && cat /var/run/reboot-required

Windows Patching Options

Windows Update · Intune · WSUS

  • Windows Update
  • Microsoft Intune
  • WSUS
  • Microsoft Configuration Manager
  • Approved business patching tool

Example Deployment Record

Patch Record · Deployment Notes

Asset: LAB-UBUNTU01
Patch Window: Saturday 12:00-14:00
Command Used: sudo apt update && sudo apt upgrade
Result: Updates installed successfully
Reboot Required: Yes
Reboot Completed: Yes
Issues Found: None
Status: Patched

Step 7

Validate The Result

Validation confirms the update installed correctly and the system still works. Do not close the patch item until validation is complete.

Ubuntu Validation

Package Check · Kernel Check · Service Check

apt list --upgradable

dpkg -l | grep openssl

uname -r

systemctl --failed

Windows Validation

Hotfix Check · Build Check

Get-HotFix | Sort-Object InstalledOn -Descending | Select-Object -First 10

Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber

Example Validation Record

Validated · Evidence Saved

Patch Item: OpenSSL security update
Asset: LAB-UBUNTU01
Validation Method: Package check and service check
Result: Update installed and no failed services found
Evidence: dpkg output and systemctl output saved
Status: Validated

Step 8

Handle Failed Or Delayed Patches

Failed and delayed patches should stay visible until they are fixed, accepted as risk, or reduced with another control.

Common Issues

  • Patch fails to install.
  • System needs a reboot.
  • Service fails after patching.
  • Disk space is too low.
  • Repository or network issue blocks the update.

Temporary Controls

  • Restrict network access.
  • Disable vulnerable service.
  • Increase monitoring.
  • Apply a configuration workaround.
  • Set a review date.

Example Deferred Patch Record

Deferred · Risk Tracked

Patch Item: Application security update
Asset: LAB-SERVER01
Reason Not Applied: Compatibility issue
Temporary Control: Restricted access to internal subnet
Owner: Lab
Review Date: 2026-07-27
Status: Deferred

Step 9

Update The Patch Tracker

A patch tracker gives a clear view of what is open, scheduled, patched, validated, failed, or deferred.

Example Tracker Entry

PATCH-001 · Validated

ID: PATCH-001
Asset: LAB-UBUNTU01
Patch Type: Security update
Package or KB: OpenSSL update
Priority: High
Owner: Lab
Status: Validated
Evidence: dpkg output and service check
Notes: No issues after reboot

Open

Patch identified but not planned yet.

Scheduled

Patch planned for a maintenance window.

Patched

Patch installed but not validated yet.

Validated

Patch confirmed and system checked.

Evidence

Evidence To Save

Save enough evidence to show what was checked, what changed, and how the result was validated.

Before Patching

  • Asset inventory entry
  • Pre-patch OS version
  • Pre-patch update list
  • Missing update finding
  • Patch priority decision
  • Change plan

After Patching

  • Patch command output
  • Reboot confirmation
  • Post-patch version check
  • Service health check
  • Scanner result if used
  • Patch tracker entry

Common Mistakes

Things To Avoid

Patch Management Mistakes

Operational Risk · Poor Evidence · Missed Validation

  • Patching without knowing which assets are in scope.
  • Skipping snapshots before risky updates.
  • Ignoring reboot requirements.
  • Marking patches complete without validation.
  • Not recording failed patches.
  • Delaying patches without a reason or review date.
  • Saving screenshots but no structured notes.
  • Not comparing before and after results.

Summary

Quick Workflow

  1. List the assets in scope.
  2. Check the current patch status.
  3. Identify missing updates.
  4. Prioritise patches by risk.
  5. Plan the change.
  6. Take a snapshot or backup.
  7. Apply the patch.
  8. Validate the result.
  9. Record failed or delayed patches.
  10. Update the patch tracker.
  11. Save evidence.

Why It Matters

Why Patch Management Matters

Patch management reduces the risk of known vulnerabilities being abused. It also supports system stability, change control, vulnerability management, and reliable IT operations.
