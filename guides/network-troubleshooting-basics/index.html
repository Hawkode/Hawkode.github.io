Guide

Network Troubleshooting Basics

This guide covers a simple step-by-step process for checking common network issues. It is written for 1st/2nd line support, infrastructure practice, and basic troubleshooting.

The goal is to avoid guessing. Start with the basics, collect evidence, test each layer, and document what you find.

Step 1

Confirm the issue

Before running commands, confirm what the user or device is actually experiencing. A clear problem statement makes troubleshooting easier.

Can the user access any websites? This helps separate full network loss from one website or service issue.

Is the issue wired, Wi-Fi, or both? This helps identify if the problem is with a cable, wireless, adapter, or wider network.

Is anyone else affected? One user affected usually means local device issue. Multiple users may mean switch, VLAN, DHCP, DNS, or internet issue.

When did it start? Recent changes can point to updates, cable moves, port changes, password changes, or service outages.

Step 2

Check physical connection or Wi-Fi

Always start with the simple checks. Many network issues are caused by loose cables, wrong ports, disabled adapters, or poor Wi-Fi signal.

Wired connection: check the Ethernet cable, wall socket, docking station, and switch port if accessible.

Wi-Fi connection: check if Wi-Fi is enabled, connected to the correct SSID, and has a usable signal.

Docking station: disconnect and reconnect the dock, then check if the network adapter appears again.

Wall port: if the device was moved, confirm the wall socket is patched correctly.

Step 3

Check IP configuration

On Windows, use ipconfig to check the current IP address, subnet mask, default gateway, and DNS servers. 

C:\Users\Hawkode> ipconfig

Ethernet adapter Ethernet:

   IPv4 Address . . . . . . . . . . : 192.168.1.25
   Subnet Mask . . . . . . . . . . : 255.255.255.0
   Default Gateway . . . . . . . . : 192.168.1.1

A normal home or office device usually has an IP address, subnet mask, and default gateway.

If the device has an address starting with 169.254, it usually means the device did not receive an address from DHCP. 

IPv4 Address . . . . . . . . . . : 169.254.45.12

Step 4

Test the default gateway

The default gateway is usually the router or Layer 3 device that connects the local network to other networks.

First, find the default gateway from ipconfig, then ping it: 

C:\Users\Hawkode> ping 192.168.1.1

Reply from 192.168.1.1: bytes=32 time=2ms TTL=64

If the gateway replies, the device can reach the local network gateway.

If it fails, check the cable, Wi-Fi connection, adapter status, VLAN, wall socket, or switch port.

Step 5

Test internet connectivity

If the gateway works, test access to an external IP address. A common test is Google DNS: 

C:\Users\Hawkode> ping 8.8.8.8

Reply from 8.8.8.8: bytes=32 time=18ms TTL=117

If this works, the device can reach the internet by IP address.

If the gateway works but 8.8.8.8 fails, the issue may be upstream: firewall, router, ISP, routing, or wider network outage.

Step 6

Test DNS

DNS converts names like google.com into IP addresses. If internet by IP works but websites do not load by name, DNS may be the issue. 

C:\Users\Hawkode> nslookup google.com

Name:    google.com
Address: 142.250.200.14

If DNS fails, check the DNS server address shown in ipconfig /all. 

C:\Users\Hawkode> ipconfig /all

Common DNS-related symptoms include websites not loading, Microsoft 365 issues, domain login problems, or internal resources not resolving.

Step 7

Check DHCP

DHCP automatically gives devices their IP settings. If DHCP fails, the device may not receive a valid IP address.

A quick Windows renewal test: 

C:\Users\Hawkode> ipconfig /release
C:\Users\Hawkode> ipconfig /renew

If the device still gets a 169.254 address, check:

  • network cable or Wi-Fi connection
  • correct VLAN or switch port
  • DHCP server availability
  • DHCP scope capacity
  • network adapter status

Step 8

Check adapter status

Sometimes the issue is local to the network adapter. On Windows, check adapter status from Control Panel or Settings.

You can also use this command: 

C:\Users\Hawkode> ipconfig /all

Check for:

  • disabled adapter
  • missing IP address
  • wrong DNS server
  • incorrect default gateway
  • VPN adapter conflicts
  • docking station adapter problems

Step 9

Check VLAN, wall socket, or switch port

If the issue happens on a wired connection, the physical network path matters. A device may be connected to the wrong wall port, unpatched socket, or wrong VLAN.

Wall socket: confirm the socket number and location.

Patch panel: confirm the wall socket is patched to the correct switch.

Switch port: check if the port is active and assigned to the correct VLAN.

Documentation: use port mapping records where available.

Good port mapping and labelling reduces guesswork and speeds up troubleshooting.

Step 10

Document and escalate

Good troubleshooting notes help the next person understand what was checked. This is especially important when escalating from 1st line to 2nd line or infrastructure teams.

Issue: user cannot access internet on wired connection.

Device: laptop connected through docking station.

IP check: device received 169.254 address.

Gateway test: failed.

Physical check: cable and dock reseated, no change.

Next step: check wall socket patching and switch port VLAN.

Summary

Quick troubleshooting flow

1. Confirm the issue and scope.

2. Check cable, Wi-Fi, dock, or wall socket.

3. Run ipconfig and check IP settings.

4. Ping the default gateway.

5. Ping an external IP address such as 8.8.8.8.

6. Test DNS with nslookup.

7. Check DHCP if the IP address is missing or starts with 169.254.

8. Check adapter status, VLAN, wall socket, and switch port.

9. Document findings clearly before escalation.
