Security Lab

Splunk Security Log Analysis Lab

This guide documents a practical Splunk security log analysis lab focused on collecting, searching, and reviewing security events from Windows and Linux systems. The lab covered Splunk Enterprise setup, log forwarding, SPL searches, dashboard ideas, alert logic, and reporting.

The aim was to understand how security logs can be centralised and analysed to improve visibility into authentication activity, account changes, SSH access, system events, and possible security incidents.

Installation

Installing Splunk Enterprise

Splunk Enterprise was installed as the central platform for collecting and analysing security logs. After installation, Splunk was started and configured so it could be used as the main search and monitoring interface.

Example Installation Flow

Splunk Enterprise • Linux • Installation

# Download Splunk Enterprise from the official Splunk downloads page
# Example package name:
# splunk-<version>-Linux-x86_64.tgz

tar -xvzf splunk-<version>-Linux-x86_64.tgz -C /opt
cd /opt/splunk/bin
./splunk start --accept-license

Splunk was also configured to start automatically when the system boots. 

cd /opt/splunk/bin
./splunk enable boot-start

After Splunk was running, the web interface could be accessed through the browser. 

http://splunk-server:8000

SPL Searches

Failed Login Detection

Failed login activity was one of the main areas reviewed. Repeated failed logins can indicate incorrect saved credentials, user mistakes, brute-force attempts, or password spraying. Splunk searches were used to group events by account and source address.

Windows Failed Login Search

Windows Security Logs • EventCode 4625 • Failed Logins

index=security sourcetype="WinEventLog:Security" EventCode=4625
| stats count by Account_Name, Source_Network_Address
| sort - count

What This Search Shows

  • Accounts with failed logins.
  • Source addresses linked to failures.
  • Repeated authentication failures.
  • Possible brute-force or password spraying patterns.

Investigation Questions

  • Is one account failing repeatedly?
  • Are many accounts failing from the same source?
  • Is the source expected?
  • Did a successful login happen after many failures?

Linux Security

SSH Access Tracking

SSH access was reviewed to understand which users authenticated successfully and which source addresses were involved. This is useful for checking expected administration activity and identifying unusual remote access patterns.

SSH Successful Access Search

Linux • SSH • Authentication Logs

index=security sourcetype=linux_secure "Accepted password"
| stats count by user, src_ip
| sort - count

SSH Failed Access Search

Linux • Failed Passwords • SSH Monitoring

index=security sourcetype=linux_secure "Failed password"
| stats count by src_ip, user
| sort - count

SSH Fields Reviewed

  • User
  • Source IP address
  • Authentication result
  • Host
  • Time of access

Useful Checks

  • Unexpected source addresses.
  • Repeated SSH activity.
  • Root or admin-style access.
  • Access outside expected time windows.

Alerting

Security Alert Logic

Alert logic was included to show how searches can move from manual review to detection logic. One example was detecting repeated failed SSH login attempts from the same source or against the same user.

Example Brute Force Alert Search

Alerting • Failed Passwords • Brute Force Detection

index=security sourcetype=linux_secure "Failed password"
| stats count by src_ip, user
| where count > 5

Alert Ideas

  • Multiple failed login attempts.
  • Admin or root login from a new IP address.
  • Changes to sensitive files such as /etc/passwd.
  • Repeated failed SSH activity.
  • Unusual account changes.

Alert Review

  • Check whether the source is expected.
  • Review the affected account.
  • Check for successful login after failures.
  • Confirm whether the event is normal lab activity.
  • Document the investigation decision.

Lessons Learned

Key Takeaways

This lab showed how Splunk can centralise security events and turn raw logs into useful investigation data. The key skill is not only collecting logs, but knowing how to search, filter, summarise, and document what the events mean.

Technical Takeaways

  • Splunk Enterprise can collect and search security logs centrally.
  • Universal Forwarder can send Windows events into Splunk.
  • Linux syslog and auditd add useful visibility.
  • SPL searches can reveal repeated failures and account activity.
  • Dashboards make recurring review easier.

Security Takeaways

  • Failed logins need context before conclusions are made.
  • SSH activity should be reviewed for unusual sources.
  • Account changes are important security events.
  • Alerts need thresholds to reduce noise.
  • Reporting turns technical logs into useful records.
